# Formatter Agent

## Role
Assembles structured data into final markdown output. Supports all 4 modes: changelog (3 templates), PR review, direction recommendations, and health reports.

## Inputs
- For **changelog**: Reads `references/writer_output.json` (WriterOutput schema)
- For **pr_review**: Reads `references/pr_review_output.json` (PRReviewOutput schema)
- For **direction**: Reads `references/direction_output.json` (DirectionRecommendation schema)
- For **health**: Reads `references/community_metrics.json` (CommunityMetrics schema)
- Also reads `references/orchestrator_state.json` for `{ mode, owner, repo }`

## Outputs
Writes `references/formatted_output.json` conforming to `FormatterInput` schema — a mode-wrapped object with `rawMarkdown`.

## Mode: changelog

### Step 1: Build the Release Header

```markdown
# 🎉 Release {toRef}

**{dateRange}** · {totalCommits} commits · {filesChanged} files · +{additions}/-{deletions} lines

✨ **{totalCommits}** commits · 👥 **{contributors.length}** contributors · 🏆 **{topContributor}** led

---
```

### Step 2: Build Community Highlights (All Templates)

```markdown
## 🌟 Community Highlights

{for each highlight}
### [{title}]({url})
> Requested by **{requestedBy}** · 👍 {reactions} · 💬 {comments}

{description}

{communityImpact}
{/for each}
```

### Step 3: Build Categorized Changes

```markdown
{for each category section}
## {emoji} {title}

{for each item}
- **{description}** {if userBenefit}— *{userBenefit}*{/if}
  {if link} [↗]({link}){/if} — by {author}
{/for each}
{/for each}
```

### Step 4: Build Roadmap

```markdown
## 🗺️ Upcoming

| Feature | Reactions | Likelihood |
|---------|-----------|------------|
{for each roadmap item}
| [{title}]({url}) | {reactions} 👍 | {likelihood} {if prediction}— {prediction}{/if} |
{/for each}
```

### Step 5: Build Footer (Template-Specific)

**Community:**
```markdown
---
*Generated by changelog-release-notes skill • Release: {toRef}*
```

**Technical:**
```markdown
---
**Release {toRef}** · [`{fromRef}...{toRef}`]({compareUrl}) · Generated by changelog-release-notes
```

**Marketing:**
```markdown
---
📦 [Download {toRef}]({releaseUrl}) · 🐛 [Report Issue](https://github.com/{repo}/issues/new) · ⭐ [Star on GitHub](https://github.com/{repo})

*Made with ❤️ by the {owner} team and contributors*
```

### Template Style Rules

| Aspect | Community | Technical | Marketing |
|--------|-----------|-----------|-----------|
| Tone | Warm, Chinese-first | Precision, English, data-driven | Punchy, benefit-focused |
| Header | "发布说明" | "Release Notes" | "What's New" |
| Emoji use | Moderate | Minimal | Heavy (highlights section titles) |
| Contributor mention | Yes, highlighted | Yes, listed | Yes, in footer |
| Issue links | Inline | Reference-style footnotes | Inline |
| `userBenefit` | Optional italic | Not shown | Required italic after description |
| `communityImpact` | Always | Only if relevant | Always |

## Output JSON Structure

The formatter MUST output data conforming to `FormatterInput` schema — all mode-specific data is nested, and `rawMarkdown` is at the top level:

```typescript
// For changelog mode:
{
  "repo": "owner/repo",
  "mode": "changelog",
  "writerOutput": { /* full WriterOutput object */ },
  "rawMarkdown": "# 🎉 Release ..."
}

// For pr_review mode:
{
  "repo": "owner/repo",
  "mode": "pr_review",
  "prReviewOutput": { /* full PRReviewOutput object */ },
  "rawMarkdown": "## PR Review: #42 ..."
}

// For direction mode:
{
  "repo": "owner/repo",
  "mode": "direction",
  "directionOutput": { /* full DirectionRecommendation object */ },
  "rawMarkdown": "## Direction Recommendations ..."
}

// For health mode:
{
  "repo": "owner/repo",
  "mode": "health",
  "healthOutput": { /* full CommunityMetrics object */ },
  "rawMarkdown": "## Health Report ..."
}
```

## Implementation

```typescript
function buildFormatterInput(mode, data, rawMarkdown): FormatterInput {
  const base = { repo: `${data.owner}/${data.repo}`, mode, rawMarkdown };

  switch (mode) {
    case "changelog":
      return { ...base, writerOutput: data };
    case "pr_review":
      return { ...base, prReviewOutput: data };
    case "direction":
      return { ...base, directionOutput: data };
    case "health":
      return { ...base, healthOutput: data };
  }
}
```

## Important Rules

1. **Markdown validity** — Every `[text](url)` must have a valid URL. No bare brackets.
2. **Item count limits** — No section should exceed 10 items. If more than 10, show top 10 and add "...and {n-10} more".
3. **No empty sections** — If data is empty, omit the entire section.
4. **Header images** — Do NOT include images or base64-encoded assets. Text-only markdown.
5. **Line wrapping** — Keep markdown line lengths under 120 characters.
6. **Escaped characters** — Escape `|` in table content, escape `*` in markdown text.
7. **Mode check** — Always read `orchestrator_state.json` mode before formatting. Use the correct template for each mode.
8. **Output nesting** — Always wrap mode-specific data inside the correct field (`writerOutput`, `prReviewOutput`, `directionOutput`, `healthOutput`). Do NOT flatten.
